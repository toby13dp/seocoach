     9→## Overview
    10→
    11→This document defines the complete functional requirements for SEOCoach, an AI-driven SEO automation platform for Dutch entrepreneurs, marketers, e-commerce businesses, and SEO agencies. Requirements are organised by module and mapped to implementation phases. Every requirement must be traceable in `REQUIREMENTS_TRACEABILITY.md`.

    15→## Module 1: Foundation & Multi-Tenant Platform
    16→
    17→### 1.1 Authentication
    18→
    19→| ID | Requirement |
    20→|----|------------|
    21→| AUTH-001 | Users can register with email and password. Registration requires email verification. |
    22→| AUTH-002 | Users can log in and log out. Sessions are managed securely with configurable expiry. |
    23→| AUTH-003 | Password reset architecture is implemented. Passwords are hashed using a modern, adaptive algorithm (bcrypt or Argon2). |
    24→| AUTH-004 | Rate limiting is enforced on authentication endpoints to prevent brute-force attacks. |
    25→| AUTH-005 | Authentication audit events are recorded: registration, login, logout, failed login, password reset, email verification. |
    26→| AUTH-006 | Development mode uses a local email preview mechanism. No paid email provider is required. |
    27→| AUTH-007 | Session management supports concurrent session control and remote session termination. |

    29→### 1.2 Multi-Tenancy
    30→
    31→| ID | Requirement |
    32→|----|------------|
    33→| TEN-001 | Organisations are the primary tenant boundary. Every tenant-owned database record includes an explicit `organisationId`. |
    34→| TEN-002 | Organisation memberships link users to organisations with defined roles. |
    35→| TEN-003 | Clients represent agency clients within an organisation. |
    36→| TEN-004 | Website projects belong to a client and contain domains, locations, and all project data. |
    37→| TEN-005 | Domains are registered within a project and validated before crawling. |
    38→| TEN-006 | Locations represent physical or service-area locations for local SEO within a project. |
    39→| TEN-007 | User invitations are sent by organisation owners or managers. Invitation links expire. |
    40→| TEN-008 | Every tenant-owned query is tenant-aware: database queries include the tenant context and cannot accidentally return cross-tenant data. |
    41→| TEN-009 | Every background job carries tenant context and operates only within that tenant's scope. |
    42→| TEN-010 | Automated tests prove that one organisation cannot read or modify another organisation's data. |

   126→## Module 2: Crawling & Technical SEO
   127→
   128→### 2.1 Safe Crawler
   129→
   130→| ID | Requirement |
   131→|----|------------|
   132→| CRAWL-001 | The crawler respects domain allowlists: only explicitly approved domains are crawled. |
   133→| CRAWL-002 | The crawler honours robots.txt directives including crawl-delay. |
   134→| CRAWL-003 | Crawl rate limiting is configurable per project. Maximum page limits and depth limits prevent runaway crawls. |
   135→| CRAWL-004 | URL normalisation prevents duplicate crawling of equivalent URLs. |
   136→| CRAWL-005 | Sitemap discovery and XML sitemap parsing are supported. |
   137→| CRAWL-006 | Redirect handling follows redirects within limits; redirect chains and loops are detected. |
   138→| CRAWL-007 | HTML parsing extracts: title, description, headings, canonical, meta robots, language, internal links, external links, images, structured data, and main content. |
   139→| CRAWL-008 | Crawl progress is visible to the user. Crawls can be cancelled. |
   140→| CRAWL-009 | SSRF protection blocks: internal IP addresses, localhost, cloud metadata endpoints, non-HTTP protocols, and redirects to private networks. |
   141→| CRAWL-010 | Protection against: excessively large responses, decompression bombs, and malicious HTML sanitisation. |
   142→| CRAWL-011 | A Playwright rendering mode is available for selected pages. Not every page is rendered by default. |
   143→| CRAWL-012 | Page snapshots are stored for each crawl, enabling historical comparison. |
   144→| CRAWL-013 | Crawl history is maintained with full metadata: start time, end time, pages found, pages crawled, errors, and configuration. |
   145→
   146→### 2.2 Content Inventory
   147→
   148→| ID | Requirement |
   149→|----|------------|
   150→| INV-001 | A searchable page inventory displays all crawled pages with filters, sorting, and pagination. |
   151→| INV-002 | Bulk selection enables actions across multiple pages. Saved views persist filter combinations. |
   152→| INV-003 | Each page record stores: URL, status, title, description, H1, word count, canonical, indexability, content type, language, publication date, modification date, internal links, external links, images, alt text, structured data, duplicate similarity, crawl depth, and conversion elements. |
   153→| INV-004 | Export to CSV and other formats is supported. |
   154→| INV-005 | Simple and technical page detail views are available. The simple view hides technical details by default. |
   155→
   156→### 2.3 Technical SEO Rule Engine
   157→
   158→| ID | Requirement |
   159→|----|------------|
   160→| RULE-001 | An extensible rule engine evaluates crawled pages against a comprehensive set of SEO rules. |
   161→| RULE-002 | Rules cover: 4xx/5xx pages, redirect chains, redirect loops, missing canonical, conflicting canonical, canonical to error, noindex conflicts, robots.txt blocking, sitemap conflicts, missing title, duplicate title, weak/unclear title, missing description, duplicate description, missing H1, multiple H1, heading structure, broken internal links, broken external links, orphan pages, deep pages, missing image alt text, oversized images, mixed content, HTTPS problems, structured data errors, hreflang errors, pagination issues, thin content, duplicate content, near-duplicate content, shadow content, staging URLs, parameter explosion, faceted navigation risks, and JavaScript rendering differences. |
   162→| RULE-003 | Every issue includes: Dutch plain-language explanation, technical details, evidence, severity, priority, impact, effort, affected URLs, recommended action, possible automatic fix, and confidence level. |
   163→| RULE-004 | New rules can be added without modifying core engine code (plugin architecture). |

   167→| ID | Requirement |
